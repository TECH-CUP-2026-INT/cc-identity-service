# Introduction

The **CC Identity Service** is the authentication and authorization backbone of the TechCup platform. It owns user credentials and is the only service allowed to issue the JWT tokens that protect every other microservice.

## Purpose

TechCup digitizes the semester football tournament at Escuela Colombiana de Ingeniería Julio Garavito. Every actor — students, guests, graduates, captains, referees, organizers and administrators — must authenticate before interacting with the platform. The Identity Service provides that single, secure entry point.

## Identity model

A `User` is identified by a UUID (generated by `users-players-service`, the source of truth for profile data). The Identity Service stores only authentication-related fields:

| Field | Description |
|-------|-------------|
| `id` | UUID, matches the user in `users-players-service` |
| `email` | Institutional email (`@escuelaing.edu.co`) |
| `password` | BCrypt-hashed password |
| `userType` | `STUDENT`, `GUEST` or `GRADUATE` |
| `role` | `PLAYER`, `CAPTAIN`, `REFEREE`, `ORGANIZER` or `ADMIN` |
| `status` | `ACTIVE`, `INACTIVE` or `LOCKED` |
| `failedLoginAttempts` / `lockedUntil` | Account lockout tracking |

## Authentication flow

1. The user posts credentials (or a Google token) to `/api/v1/auth/login`.
2. If valid, an OTP is emailed and a `userId` is returned — the session is **not** yet authenticated.
3. The user posts the OTP to `/api/v1/otp/validate` and receives a JWT plus profile data.
4. Other services validate that JWT via `/api/v1/token/validate`.
5. Logout revokes the token via `/api/v1/auth/logout`.

## Where it fits

- **Issues** JWTs consumed by `cc-tournament-service`, `cc-teams-service`, `cc-users-players-service` and `am-comunication-service`.
- **Consumes** `cc-users-players-service` for live role/status (via OpenFeign) and `am-notification-service` for email delivery.
- **Exposes** internal endpoints for `cc-users-players-service` (credential lifecycle) and `am-notification-service` (email resolution).
